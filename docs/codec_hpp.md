# Codec Layer (`include/nam/codec.hpp`)

## Overview

The codec layer implements one of the central conceptual claims of the NAM
(Number-as-Machine) framework: **the numeric base is a codec, not an intrinsic
property of a number** (Section 3.5). A number is an abstract object; its
radix expansion is merely one of many possible *projections* of that object
into a stream of digits. Changing the base changes the projection — the lens
through which we view the number — but it does not change the number itself.

This module provides the machinery to *reproject* a number from a source base
into a target base, treating base conversion as a re-decoding operation rather
than an arithmetic transformation of stored digits.

## Background and Theory

### Positional Numeral Systems and Radix Representation

The idea that a number admits many radix representations is foundational to
number theory and computer arithmetic. A real value `x ∈ [0,1)` has a
base-`b` expansion

> x = Σ_{i≥1} d_i · b^{-i}, d_i ∈ {0, …, b−1}

The digits `d_i` are not the number; they are the output of a *decoding map*
parameterised by `b`. This perspective is well established in the study of
**β-expansions** (Rényi, 1957; Parry, 1960), where representations of reals in
non-integer and integer bases are analysed as dynamical systems generated by
the map `T(x) = bx mod 1`. The repeated multiply-by-base extraction used here
(`reproject_digits` and the `Rational` VM) is precisely the orbit of `x` under
this transformation.

### Base as Projection, Not State

Classical representations (e.g. arbitrary-precision integers) bake the base
into stored data. By contrast, NAM follows the **stream / corecursion** view of
numbers (cf. exact real arithmetic via streams, Bertot & Komendantsky, 2008;
Edalat & Potts, 1997). Here a number is a *coalgebraic* object: a generator
that, on demand, emits its next digit. Under this view a base change is a change
of the observation morphism, formalising the slogan *"base is a codec"*.

This aligns with **exact real arithmetic** traditions, where reals are
represented by infinite digit streams (signed-digit, redundant, or continued
fraction representations) and operations are defined as stream transducers
rather than operations on fixed-width data (Vuillemin, 1990; Plume, 1998).

### Interval-Honest Reprojection

A faithful conversion between bases must respect the fact that reading a finite
prefix of source digits only pins the value to an *interval*, never an exact
point. The implementation is **interval-honest**: reading `k` source digits
yields an exact rational lower bound `num/den` with `den = base^k`, which
ignores the (bounded) tail contribution. This mirrors the methodology of
**interval arithmetic** (Moore, 1966) and the precision-tracking discipline of
exact real computation, where every finite observation carries an explicit
error bound.

### Rationals Under Base Change

For rational sources the conversion is analytic rather than streaming. A
rational `p/q` retains its identity under any base; only its digit projection
changes. This is the cleanest instance of the codec principle, and connects to
the classical result that a rational's expansion is eventually periodic in
*every* integer base — a corollary of the orbit of `T(x) = bx mod 1` being
finite on rationals (the periodicity / cyclic-number results going back to
Gauss). We exploit this by simply re-seeding the rational generator with the
new base.

## API Reference

### `rational_in_base(AutomatonVM rat, uint32_t new_base)`

Analytic reprojection for rationals. A rational keeps its identity under a base
change, so the VM is rebuilt with the new codec by re-seeding `make_rational`
with the original numerator and denominator (`state[2] = p`, `state[1] = q` as
laid down by `make_rational`) and the new base.

| Parameter   | Description                                                   |
|-------------|---------------------------------------------------------------|
| `rat`       | A rational `AutomatonVM` previously built by `make_rational`. |
| `new_base`  | The target base for the new projection.                       |
| **Returns** | A new `AutomatonVM` projecting the same value in `new_base`.  |

This is the canonical demonstration that *base is a codec*: identity preserved,
projection swapped.

### `reproject_digits<G>(AutomatonVM src, uint32_t target_base, int src_digits, int out_digits)`

Generic streaming reprojector. Reads a generator `src` (producing digits in
`src.base` for a value in `[0,1)`) and emits digits in `target_base`.

**Algorithm:**

1. Read `src_digits` source digits into an exact rational fraction `num/den`
   where `den = src.base^src_digits`. This `num/den` is an exact *lower bound*
   on the value (the unread tail is discarded), keeping the conversion
   interval-honest.
2. Emit `out_digits` digits of `num/den` in `target_base` by driving the
   `Rational` VM (the orbit of `T(x) = target_base · x mod 1`).

| Parameter     | Description                                                       |
|---------------|-------------------------------------------------------------------|
| `G`           | Generator concept type whose `step` advances the source VM.       |
| `src`         | Source `AutomatonVM` (value in `[0,1)`).                          |
| `target_base` | Base of the emitted digit stream.                                 |
| `src_digits`  | Number of source digits to consume (sets precision of the bound). |
| `out_digits`  | Number of target-base digits to emit.                             |
| **Returns**   | `std::vector<uint32_t>` of `out_digits` digits in `target_base`.  |

**Design note (Phase boundary):** A *fully* faithful streaming reprojection
requires a growing rational accumulator (an unbounded-precision running window),
which is a Phase-2 *series-tier* concern. For Phase 1 this function is provided
as a **bounded host-side helper** — not an `AutomatonVM` step — used in tests
and by the rational fast path. The finite `src_digits` parameter makes the
precision (and hence the error bound) explicit.

## Implementation Notes

- **Exactness for round-trips.** Rational sources are reprojected analytically
  via `rational_in_base`, ensuring the round-trip property tests
  (base `b → b' → b`) recover the original value exactly. The generic streaming
  path is bounded and approximate by construction, since it truncates the source
  tail.
- **Overflow considerations.** `reproject_digits` accumulates `num` and `den`
  in `uint64_t`. The caller must choose `src_digits` so that
  `src.base^src_digits` fits in 64 bits; large `src_digits` or `src.base` will
  overflow silently. This bound is the Phase-1 cost of not yet having the
  Phase-2 growing accumulator.
- **Dependencies.** Relies on `make_rational` and the `Rational` generator from
  `nam/rational.hpp`, the `Generator` concept from `nam/generator.hpp`, and the
  `AutomatonVM` / `NumVMStep` types from the ABI header `nam/abi.h`.
